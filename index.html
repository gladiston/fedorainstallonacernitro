REAPROVEITANDO /HOME ANTES DE INSTALAÇÃO?

Os mais experientes em geral colocam a partição /home separada e ao reinstalar o sistema deixam essa partição sem formatação e daí reaproveitam o que já tinham nela. Essa abordagem é muito boa, contudo há um pequeno problema, porém fácil de resolver: configurações anteriores de temáticas.

As configurações de quem logou-se antes poderiam ter fontes, temas e papéis de parede que antes existiam, mas agora que instalou o sistema não existem mais, essa configuração fica orfão dos arquivos e programas que antes existiam. Por exemplo, talvez a fonte de terminal tenha sido xxxxxxx e agora que o sistema foi reinstalado a fonte xxxxxxx ainda não está instalada e como resultado ao abrir o terminal surge uma fonte esquisita que não é a default do sistema e tampouco a xxxxxxxx. Para lidar com este problema, antes de formatar o sistema eu costume saber qual era a aparência original anterior com os comandos:

  • gsettings get org.gnome.desktop.interface document-font-name

'Sans 11'

  • gsettings get org.gnome.desktop.interface font-name

'Ubuntu 11'

  • gsettings get org.gnome.desktop.interface monospace-font-name

'Ubuntu Mono 13'

  • gsettings get org.gnome.desktop.background picture-uri

'file:///usr/share/backgrounds/warty-final-ubuntu.png'

gsettings get org.gnome.shell.extensions.user-theme name

Daí após o primeiro login no sistema poderemos restaurar o original que vem com o sistema e assim resolvemos possíveis configurações órfãs:

  • gsettings set org.gnome.desktop.interface document-font-name 'Sans 11'
  • gsettings set org.gnome.desktop.interface font-name 'Ubuntu 11'
  • gsettings set org.gnome.desktop.interface monospace-font-name 'Ubuntu Mono 13'
  • gsettings set org.gnome.desktop.background picture-uri 'file:///usr/share/backgrounds/warty-final-ubuntu.png'
  • gsettings set org.gnome.shell.extensions.user-theme name "Flat-Remix"

e depois que as temáticas anteriormente foram instaladas, repetimos o comando acima com os nomes que salvamos antes.

Se seu login depois de instalado será gladiston e já havia um /home/gladiston antes, então renomeie para /home/gladiston.1 porque o Ubuntu remove a pasta anterior ao invés de aproveitá-la. Isso foi uma constatação que percebi ser muito ruim. A distro Manjaro por exemplo,simplesmente renomeia a /home antiga para @home e você nunca perde nada.

PRÉ-INSTALAÇÃO NO ACER NITRO

O Acer Nitro 5 vem equipado com o chamado “Secure Boot” que na prática impede que sistemas operacionais diferentes do Windows sejam instalados. Então há mudanças que precisam ser feitas aqui.

Para entrar na BIOS, a tecla F2 deve estar pressionada assim que o computador é ligado.

A primeira coisa a se fazer é criar uma senha de supervisor ("Set Supervisor Password" que está na guia “Security”, depois disso tecle F10 para salvar e sair.

Entre na BIOS novamente, depois de fornecer a senha siga a sequência:

1- Na guia “Main” o “SecureBoot” deve estar desabilitado

2- Deve setar isso na base do UEFi

3- Na guia “Main” ajuste “SataMode” para “AHCi”, se essa opção não aparece, pressione Ctrl+S e ela aparecerá.

4- Na guia “Main” habilite a opção de “F12 Boot Menu”.

5- Na guia “Main” desabilite a opção de “FastBoot” para quando for instalar o sistema por meio de pendrive. Essa opção habilita a tecla “F12” para escolher por onde iniciar o sistema, sem ela não daria para instalar o sistema a partir dum pendrive.  Depois de instalar o sistema poderá habilitá-la novamente para ter um boot mais rápido.

6- Na guia Boot desabilite o Security Boot.

Depois dessas mudanças tecle F10 para salvar e sair.

Agora está pronto para iniciar a instalação num Acer Nitro 5, assim que ligar o notebook e aparecer o logo da Acer pressione F12 e escolha iniciar pelo pendrive e após isso, dará boot pelo pendrive e bastará seguir as instruções na tela para instalar o novo sistema operacional.

ADVERTÊNCIA: Tenha certeza de que o “Security BIOS” foi desativado, caso contrário terá problemas em alguns passos. Depois de ter tudo concluído poderá ativá-lo novamente.

Ao instalar o Linux, escolha não instalar o driver proprietário NVIDIA durante a instalação, no final do processo será instalado o driver nouveau, um driver livre para as placas nVIDIA. Isso foi importante nos últimos dois anos!

Habilitando o Delta RPM

O Fedora Linux é um comedor de atualizações, depois de instalado ele provavelmente terá mais de 1 GB de atualizações e isso é considerado normal. Mas há um recurso conhecido como deltaRPM que permite fazer as atualizações de maneira mais economica funciona assim, imagine um programa de 100MB, se este programa sofrer uma atualização, ao invés de baixarmos de novo mais 100MB baixaremos apenas um remendo(patch) contendo o que é preciso mudar no programa original para ele ser exatamente igual a versão mais recente. Os patches em geral são pequenos.

As atualizações podem economizar até 80%, mas se isso é bom então porque não é ativado por padrão? Porque perde-se algum tempo aguardando cada patch ser aplicado. Se seu computador for rapido, apliacar um patch pode não ser um problema, mas se seu computador for lento demais compensaria mais esperar pelo download inteiro do que aplicar um patch. Então o deltaRPM é uma troca, em troca de downloads mais rapido você abdica de uma atualização posterior mais lenta - simples assim.

E para habilitar esse recurso no Fedora é bem simples, abra um terminal e instale o pacote "deltarpm":

  • sudo dnf install deltarpm vim

Depois editamos o arquivo /etc/dnf/dnf.conf:

  • sudo vim /etc/dnf/dnf.conf

Adicione a linha:

deltarpm=1

Salve-o. Depois execute no terminal:

  • sudo dnf update
  • sudo dnf upgrade

As atualizações do fedora são assustadoramente pesadas, prepare-se para uma primeira atualização na casa do 1GB, mas graças ao recurso de deltaRPM esse download até 80% menor, mesma que patches sejam mais lentos, em máquinas novas isso realmente recompensa velocidades de internet menores.

DRIVER INTEL OU NVIDIA COMO GPU PRIMARIA?

Podemos usar driver INTEL como primário e deixar o driver NVIDIA como GPU para outros fins. Esse cenário é mais econômico do que usar NVIDIA como GPU primária.

Algo essencial é descobrir, qual a placa de vídeo que temos em mãos, execute:

Primeiro verifique qual é a sua placa de vídeo, execute:

  •  lspci | grep VGA

00:02.0 VGA compatible controller: Intel Corporation CoffeeLake-H GT2 [UHD Graphics 630]

01:00.0 VGA compatible controller: NVIDIA Corporation TU117M [GeForce GTX 1650 Mobile / Max-Q] (rev a1)

Em notebooks como o Nitro há duas placas de vídeo, precisamos saber qual delas operará como primária, isto é, que será usada pelo sistema operacional e a tela do notebook, então executamos:

  • glxinfo | egrep "OpenGL vendor|OpenGL renderer"

E uma saída como essa é esperada:

OpenGL vendor string: Intel

OpenGL renderer string: Mesa Intel(R) UHD Graphics 630 (CFL GT2)

Indicando que a placa Intel faz a renderização primária, o que é aceitável, pois consome menos bateria. Em alguns momentos vamos querer usar a placa nVIDIA para executar aplicações de forma mais acelerada.

REJEITANDO O NOVEAU

É interessante saber que o driver binário nvidia tem problemas com o driver noveau - alternativa opensource. Recomendo que o driver noveau seja desabilitado. Primeiro, crie o arquivo:

  • sudo vim /usr/lib/modprobe.d/blacklist-nouveau.conf

E adicione as linhas:

blacklist nouveau

options nouveau modeset=0

Salve o arquivo e saia do editor.

Edite também o  /etc/default/grub:

  • sudo vim /etc/default/grub

Procure a linha:

GRUB_CMDLINE_LINUX=”…”‘

e adicione ao final dessa linha, pouco antes de fechar aspas:

rd.driver.blacklist=nouveau nvidia-drm.modeset=1

Ficando mais ou menos assim:

GRUB_CMDLINE_LINUX="rhgb quiet rd.driver.blacklist=nouveau nvidia-drm.modeset=1"

Agora vamos atualizar o grub2 conf, execute:

  • sudo grub2-mkconfig -o /boot/grub2/grub.cfg

Então removemos o pacote xorg-x11-drv-nouveau:

  • sudo dnf remove xorg-x11-drv-nouveau

Agora falta gerar um novo initramfs, mas antes, vamos fazer um backup do antigo:

  • sudo mv /boot/initramfs-$(uname -r).img /boot/initramfs-$(uname -r)-nouveau.img

Para depois gerarmos um novo initramfs:

  • sudo dracut /boot/initramfs-$(uname -r).img $(uname -r)

        

e depois gere o próximo boot (initramfs) sem a carga deste driver:

  • sudo dracut --force

(isso pode demorar um pouco simplesmente aguarde)

Então, reinicie o computador ao final do processo, isso é mandatório.

PARTIÇÕES NTFS NO SISTEMA

Se utiliza uma partição Windows (NTFS) para gravar seus arquivos e dados a partir do Linux, você pode simplesmente não fazer nada e usar o gerenciador de arquivos do GNOME para entrar e sair do disco NTFS quando quiser. Contudo, se você tem que ir para o terminal e acessá-lo dali então lhe seria conveniente criar uma pasta vazia que ao entrar nela você já observasse o conteúdo dessa partição, se você gostou da idéia então vamos implementá-la:

Identifique qual é o seu disco, execute no terminal:

  • sudo blkid |grep ntfs
  1. /dev/nvme1n1p2: LABEL="Windows" BLOCK_SIZE="512" UUID="389083EB9083AE46" TYPE="ntfs" PARTLABEL="Basic data partition" PARTUUID="f8ff4bca-3ef8-4ba7-b1b1-6e0b00689aab"
  2. /dev/nvme1n1p3: LABEL="DADOS" BLOCK_SIZE="512" UUID="1EB4CCF2B4CCCE09" TYPE="ntfs" PARTLABEL="Basic data partition" PARTUUID="0acfc054-4ac1-46f6-83e3-c90bb5e79f12"

No exemplo anterior, o disco desejado tem o LABEL=DADOS e UUID="1EB4CCF2B4CCCE09" , então vamos criar uma pasta vazia, preferencialmente com o nome da label do disco na $HOME ou /media (ou /mnt), ex: /home/usuario/label_dados ou /media/label_dados, a vantagem de ficar em $HOME é que com os cuidados necessários apenas você terá acesso a ela, mas se prefere disponibilizar a partição para todos então tente:

  • sudo mkdir -p /media/label_dados
  • sudo chmod 1777 /media/label_dados
  • sudo chmod +s /bin/ntfs-3g
  • sudo usermod -aG disk $USER

Estamos deixando claro que /media/label_dados estará disponível a todos os usuários. Se escolher criar uma pasta em $HOME, não há a necessidade das linhas acima, apenas criar a pasta vazia.

Execute no terminal:

  • sudo vim /etc/fstab

Agora vamos editar o arquivo /etc/fstab e então acrescentar após a última linha:

  1. # Minha partição NTFS de label “DADOS”
  2. UUID="1EB4CCF2B4CCCE09" /media/label_dados ntfs-3g windows_names,nosuid,nodev,nofail,rw,user,gid=users,noauto 0 0

Explicando os demais parâmetros:

nosuid: Bloqueia a operação de bits suid e sgid

nodev: Não interpreta dispositivos especiais de bloco no sistema de arquivos.

nofail: faz com que o dispositivo seja montado mesmo que tenha sido marcado com erro. Um drive é marcado com ‘erro’ quando o computador é desligado abruptamente ou quando qualquer unidade listada no fstab não está presente ou falhou na montagem.

zero e zero no final da linha: todo

Uma outra forma de escrever essa linha no fstab seria:

  1. UUID="1EB4CCF2B4CCCE09" /media/label_dados ntfs nls-utf8,rw,nosuid,nodev,nofail 0 0

A diferença é que ao usar driver “ntfs-3g” você estará usando um driver do tipo userspace(pacote ntfs-3g precisa estar instalado) considerado o método mais moderno e seguro, enquanto “ntfs” está ligado a um módulo diretamente no kernel do linux que geralmente recomendamos usar apenas no modo de leitura que impede qualquer gravação na unidade. Mas pode-se habilitar o modo de leitura e escrita se desejar, já testei ambos no modo leitura/escrita e prefiro o “ntfs-3g” que além de ser mais seguro, possui outros parâmetros que nos ajudam a evitar nomes de arquivos que podemos criar usando Linux, mas que o Windows terá problemas com eles, por exemplo o uso de caracteres como  “:” e “?” em nomes de arquivos.

Salve e após o boot, a pasta indicada servirá de acesso a partição NTFS.

Se você não quiser auto montá-la no boot, mas mantê-la apenas quando executar no terminal:

  • mount /media/label_dados

Então troque “auto” para “noauto”. O “noauto” é mais seguro por impedir que programas instalados ou scripts maliciosos tenham acesso a esta partição ou disco.

Caso precise reparar a unidade NTFS, execute:

  • sudo ntfsfix /dev/disk/by-uuid/34F84B57F84B1690

ou se souber o label do disco:

  • sudo ntfsfix /dev/disk/by-label/DADOS

Alternativas: Existe um serviço chamado AutoFS, ele implementa uma solução onde você indica pastas e apenas quando você acessá-las, ele as monta. Serve para discos externos, partições internas e também para compartilhamentos remotos. Esta última, é o motivo pelo qual é mais usado visto que auto-montar pastas que já estão em nosso domínio é mais fácil usando o fstab. AutoFS é um pouco mais complicado que usar /etc/fstab, mas nem tanto depois que você entende como ele funciona. Eu tenho receio de utilizá-lo em ambientes com pouco controle porque se houver programas que vasculhem discos eles irão montar todas as pastas que encontrarem na configuração para auto montar, talvez  voce pense na situação de vírus de computador, mas ocorreria algo similar em softwares de backups que podem erroneamente incluir pastas que não deveriam. INSTALAÇÃO DE FONTES E TEMAS A PARTIR DOS REPOSITÓRIOS DE DESENVOLVEDORES

O problema de instalar fontes, temas e papéis de parede a partir de repositórios é que elas são instalados no sistema e não na sua $HOME, se este pacote for removido o comportamento de qualquer Look&Feel que tiver ajustado dependido dele ficará orfão e será exibido de forma errática.  Então, se for usar fontes, temas e papéis de parede com o objetivo de usar na temática da área de trabalho, prefira instalá-los diretamente das páginas oficiais dos desenvolvedores geralmente hospedados no github.

Para ter este conjunto de temáticas na sua $HOME crie os seguintes diretórios:

  • mkdir -p ~/.local/share/fonts/
  • mkdir -p ~/.themes
  • mkdir -p ~/.icons

Nenhum tema pré compilado que vier por fora da distribuição terá exibido se a extensão abaixo não for instalada:

https://extensions.gnome.org/extension/19/user-themes/

Para quem usa Windows, o Look’n Feel é basicamente papel de parede, cores, cantos das janelas arredondados ou translúcidos e isso é basicamente uma coisa só. No entanto, no Linux e mais especificamente GNOME há categorias de temáticas e elas estão bem separadas entre si e o Look’n Feel é junção dos temas de cada categoria, vamos entender:

  • Tema GNOME, refere-se mais aos painéis e decorações de janelas de maneira geral, às vezes chamados de compositores.
  • Tema GTK, atualmente na versão GTK3 refere-se aos componentes de formulários que estão dentro das janelas como list boxes, time pickers, barras de progressos, edits, etc…
  • Tema Ícones, os ícones.

Às vezes a escolha de um tema GNOME+GTK+Ícones pode criar um look’n feel inconsistente e em outras oportunidades a combinação é tão perfeita que nos torna mais produtivos.

Há um Look’n Feel chamado Flat-Remix onde temos o mesmo tema em todas essas categorias, ele existe nos repositórios deb, flatpack, snaps, mas vamos instalar manualmente.

IMPORTANTE: Alguns temas para serem instalados contêm instruções como descompactar ou fazer ‘git clone’ de repositórios e depois executar comandos usando ‘sudo’ como ‘sudo make install’. Isso não é recomendado porque não sabemos o que estes scripts irão fazer ou se ficaram desatualizados. Scripts que estão desatualizados podem quebrar coisas que antes funcionavam no ambiente gráfico. Também há scripts ruins ou mal intencionados que podem fazer um belo estrago então evite instruções que façam uso de scripts  como make, make install que usam ‘sudo’.

GERENCIANDO DUAS VERSÕES OU MAIS DE JAVA

  1. Primeiro observe se você tem mais de uma versão do Java instalado com o comando:

update-alternatives --list java

/usr/lib/jvm/java-11-openjdk-amd64/bin/java

O comando acima, indicou que apenas 1 versão do java está instalado no seu ambiente e é a versão do openjdk. Mas se você instalou o “Oracle Java” saiba que ele depois de instalado não aparecerá com o uso do comando update-alternatives, isso acontece porque os mais experientes usam um programa asdf ou simplesmente exportam o path da variável JAVA_HOME para indicar o java que precisam usar. Contudo, ainda acho interessante usar o update-alternatives para essa tarefa e o passo seguinte será acrescentar o Oracle java ao sistema.

Caso você tenha encontrado o Oracle Java apenas vá para o passo final.

  1. Se a versão do Oracle Java não aparece então acrescente-o:

Primeiro descubra a localização dele, se você instalou-o por meio de .deb, vá até o local do pacote e execute:

ls -l |grep "/bin"

./usr/lib/jvm/jdk-16.0.1/bin/

(...)

Caso não tenha .deb, mas ele já se encontre instalado então use o comando:

dpkg -L  jdk-16.0.1|grep "/bin"

/usr/lib/jvm/jdk-15.0.1/bin

(...)

Embora tenha usado como referência jdk-15.0.1 o mesmo serviria para default-jdk se fosse o caso dele estar faltando no sistema, por exemplo:

dpkg -L  openjdk-11-jdk|grep "/bin"

/usr/lib/jvm/jdk-16.0.1/bin

Após descobrir o path principal que se refere a pasta “bin” então prosseguimos com os comandos abaixo onde a localização da pasta “bin” do java faltante fica evidente. Mas atenção, é bom acrescentar tanto o “java” como também o “javac”:

  1. sudo update-alternatives --install "/usr/bin/java" "java" "/usr/lib/jvm/jdk-16.0.1/bin/java" 2
  2. sudo update-alternatives --install "/usr/bin/javac" "javac" "/usr/lib/jvm/jdk-16.0.1/bin/javac" 2

O “2” no final da linha refere-se a prioridade, já temos uma versão de java instalada então a próxima seria a “2”, usar um número menor que “2” significaria substituir um java pré instalado e não desejamos isso.

  1. Agora verificamos novamente quantas versões de java existem:

update-alternatives --list java

/usr/lib/jvm/java-11-openjdk-amd64/bin/java

/usr/lib/jvm/jdk-15.0.1/bin/java

Sim, agora possuímos as duas e no passo seguinte vamos escolher qual desejamos usufruir.

Finalmente vamos trocar de versão:

Trocando a versão de java, primeiro vemos qual a versão do java corrente executando o comando:

java --version

openjdk 11.0.9.1 2020-11-04

OpenJDK Runtime Environment (build 11.0.9.1+1-Ubuntu-0ubuntu1.20.04)

OpenJDK 64-Bit Server VM (build 11.0.9.1+1-Ubuntu-0ubuntu1.20.04, mixed mode, sharing)

Agora vamos trocar de versão:

sudo update-alternatives --config java

ou para o java compiler:

sudo update-alternatives --config javac

Instalando o Virtual Machine Manager

O virt-manager (Virtual Machine Manager) é uma interface gráfica desenvolvida pela Red Hat para gerenciar máquinas virtuais através do libvirt. Ele visa principalmente as VMs do KVM, mas também gerencia as máquinas virtuais do Xen e LXC (Linux Containers).

 Para instalar o virt-manager e todas as suas dependências no Fedora, execute o comando abaixo num terminal:

sudo dnf group install --with-optional virtualization

 Inicie o serviço do libvirtd com:

sudo systemctl start libvirtd

 E para iniciar o serviço durante o boot, execute:

sudo systemctl enable libvirtd
